# Desktop Validation Checklist

Date: 2026-03-10
Product: SitePulse Studio 1.0.0

## Core program checks

- [x] Desktop shell boots with a native Electron window
- [x] Custom title bar buttons still parse and bind correctly
- [x] Bridge runtime boots from the local packaged QA engine
- [x] Renderer syntax validated with `node --check`
- [x] Main process syntax validated with `node --check`
- [x] Preload bridge syntax validated with `node --check`

## UI and shell checks

- [x] Main shell viewport is constrained to the window instead of growing past it
- [x] Left, center and right columns now have explicit `overflow-y: auto`
- [x] Active view panels keep bottom padding and scroll correctly inside the shell
- [x] Scroll gutters are stable so layout does not jump when content exceeds the viewport
- [x] Internal tool surfaces now expose explicit scroll behavior (`log`, `history`, `route explorer`, `action explorer`, `steps`, `brief`, `prompt`)
- [x] Command palette is available from the desktop shell and via `Ctrl+K`
- [x] Reports view compares the current run against baseline or previous run
- [x] UI contract test passed via `npm run desktop:test:ui`
- [ ] Final tactile pass in a live Windows session after installer update

## Audit engine checks

- [x] Desktop smoke test passed via `npm run desktop:smoke`
- [x] Native Windows build passed via `npm run desktop:build:win:staging`
- [x] QA CLI run passed via `npm --prefix qa run audit:cmd -- --config "audit.default.json" --base-url "https://example.com" --no-server --scope full --fresh --human-log`
- [x] Replay command is generated by the engine
- [x] Report JSON, markdown, issue log and assistant brief are emitted by the engine

## Functional consistency checks

- [x] Audit depth is now stored in desktop state (`signal` vs `deep`)
- [x] Report summary reads the real executed depth instead of the current toggle state
- [x] Run history stores mode and depth with each snapshot
- [x] Run comparison now tracks issue delta, SEO delta, risk delta, route delta and action delta
- [x] Baseline can be pinned from the current run or from history
- [x] UI contract exercises native audit, CMD handoff, command palette, comparison, view switching and start/stop engine controls
- [ ] Decide whether SEO findings should also contribute to the "open issues" metric or remain a separate score channel

## Known follow-up items

- The product now behaves correctly as a desktop shell, but it still needs a live visual pass on the new installer after the scroll fix.
- The desktop shell now has operational tooling beyond a passive dashboard, but the next layer should still add guided compare workflows and richer progress telemetry.
- SEO findings are tracked separately from runtime/flow issues. This is valid, but the product wording should be tightened later if you want one single universal issue count.
